# Dead Backend-Feature Cleanup Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Remove frontend type/route/enum surface in `108jobs-clean` for backend features that `api-108jobs` has already fully removed (federation/ActivityPub, voting/likes, category/person block-hide, instance allow/block, registration-application admin, post/proposal reports, saved/read/hidden actions, mention/reply notifications).

**Architecture:** `108jobs-clean`'s vendored API client at `src/lib/108jobs-client` is a local `file:` dependency — the app imports the **built** `dist/` output (via `node_modules/108jobs-client` → symlink → `package.json#main`), not the `src/` directly. So client-side edits are invisible to the app until the client package is rebuilt. Work proceeds: prune the client's dead surface first (types, exports, HTTP methods), rebuild the client, then let `tsc --noEmit` on the app surface every now-broken import so nothing is missed by grep alone, then remove the remaining app-level dead enums/strings that don't come from the client at all.

**Tech Stack:** TypeScript, Next.js 16, pnpm workspaces (local `file:` dependency, not a true pnpm workspace), `@tsoa/runtime`-decorated HTTP client, ESLint, tsc.

## Global Constraints

- Confirmed dead (no backend route in `api-108jobs/src/api_routes.rs`): federation/ActivityPub, voting/likes, category/person block+hide, instance allow/block, registration-application admin review, post/proposal/private-message report creation & review, saved/read/hidden actions, mention/reply inbox notifications (`listNotifications`, `markNotificationAsRead`, `markAllNotificationsAsRead`, `getUnreadCount`).
- **Do not touch:** `report_count` (`/account/report_count` is live), rider ratings/reviews (`UserReview*` types, `/reviews` routes), chat/presence system, `LocalUser.acceptedTerms` (repurposed for the live `/account/update-term` terms-acceptance flow, despite its stale ts-rs comment saying "registration application accepted" — confirmed live via `src/services/UserService.ts` and `src/containers/SpProfile/index.tsx`), and `src/modules/chat/store/roomsStore.ts`'s `getUnreadCount` (an unrelated same-named local chat method — do not confuse with the client's dead `getUnreadCount` HTTP call).
- Client type files under `src/lib/108jobs-client/src/types/*.ts` carry a `// This file was generated by [ts-rs] ... Do not edit this file manually.` banner. This plan intentionally hand-edits them anyway — regenerating from the Rust backend is out of scope (per the approved design spec) since there's no working ts-rs export pipeline to regenerate from in this repo. Do not add a new codegen step; just edit directly.
- Every deletion in this plan was verified against the app codebase (`grep -rn` under `src`, excluding `node_modules` and `108jobs-client` itself) to have zero live consumers before being scheduled for removal. If a task's compiler/grep check surfaces an unexpected consumer not listed here, stop and re-verify against `api-108jobs/src/api_routes.rs` rather than deleting blind.
